# 直播界面卡顿 — 前端渲染优化方案

> 症状：直播时**整个界面**卡（按钮、滚动、表格都卡），非单纯视频掉帧。
> 已验证：Chrome Performance 录制显示 **Scripting(黄) + Rendering(紫) 占主线程大头** →
> 坐实前端主线程被阻塞，与 RTMP 推拉流无关。
> 分析范围：`Dashboard.vue` + `store/alarm.js` + `AlarmPanel.vue` + 后端 `alarms.py`。

---

## 1. 根因（按主线程占用从大到小）

### 根因 A — `flashTimer` 常驻全量重绘 canvas（Rendering 主因）

`Dashboard.vue:313` 有个定时器**无条件每 500ms 运行**：

```js
let flashTimer = setInterval(() => { flashOn.value = !flashOn.value }, 500)
```

`watch(flashOn)`（`:309`）→ 每次翻转都调 `drawOverlay()` **全量重画整块 overlay canvas**
（`measureText` + `arcTo` 圆角 + fill + stroke，逐 region 计算）。
**问题**：红闪本应只在有告警时才需要，现在没告警也在每 500ms 强制重绘，
与视频每帧 `drawImage` 叠加，主线程持续有活。

### 根因 B — 告警列表无上限，越用越卡（Scripting 主因）

- 后端 `alarms.py:116`：`query.order_by(...).all()` **没有 LIMIT**，历史告警全查。
- 前端 `store/alarm.js:20`：`this.alarms.unshift(alarm)` **永不裁剪**。
- `AlarmPanel.vue` 用 `el-table` 渲染**全部行**。

直播中告警不断产生 → 数组无限增长 → 每次 push 触发响应式 diff + el-table 全表重渲染，
**越直播越卡**（正好解释"直播时才卡"）。

### 根因 C — `activeRegions` 深度 watch（次要放大器）

`Dashboard.vue:295` `watch(activeRegions, ..., { deep: true })`：告警状态每变一次就
`drawOverlay()` + 启停蜂鸣。与根因 A 的重绘叠加。

---

## 2. 修复方案（按性价比排序）

### P0-A — flashTimer 改为按需启停（消除常驻重绘）

`Dashboard.vue` 中删掉常驻定时器，改成有红色告警时才启动闪烁、无告警时停掉并静止绘制。

**改动 1**：把模块级 `let flashTimer = setInterval(...)`（`:313-315`）删除，改为受控函数：

```js
let flashTimer = null

function startFlash() {
  if (flashTimer) return
  flashTimer = setInterval(() => {
    flashOn.value = !flashOn.value
  }, 500)
}

function stopFlash() {
  if (flashTimer) {
    clearInterval(flashTimer)
    flashTimer = null
  }
  flashOn.value = true   // 复位，避免停在半透明帧
}
```

**改动 2**：在已有的 `watch(activeRegions)`（`:295`）里，根据是否有红区启停闪烁（蜂鸣同一处已有判断，复用）：

```js
watch(
  () => alarmStore.activeRegions,
  () => {
    drawOverlay()
    const hasRed = Object.values(alarmStore.activeRegions).some((s) => s === 'red')
    if (hasRed) { startBeepLoop(); startFlash() }
    else { stopBeepLoop(); stopFlash() }
  },
  { deep: true }
)
```

**改动 3**：`onUnmounted`（`:341`）里 `flashTimer && clearInterval(flashTimer)` 换成 `stopFlash()`。

> 效果：无告警时 overlay canvas **完全不重绘**，主线程 Rendering 占用大幅下降。

---

### P0-B — 告警列表设上限（前端裁剪 + 后端 LIMIT）

**前端 `store/alarm.js`**：`push` 时裁剪，historyload 时也裁：

```js
const MAX_ALARMS = 100

loadAlarms(alarms) {
  const list = Array.isArray(alarms) ? alarms.slice(0, MAX_ALARMS) : []
  this.alarms = list
  list.forEach((alarm) => {
    if (alarm.level !== 0 && alarm.status !== 'confirmed') {
      this.activeRegions[alarm.region_id] = 'red'
    }
  })
},

push(alarm) {
  this.alarms.unshift(alarm)
  if (this.alarms.length > MAX_ALARMS) {
    this.alarms.length = MAX_ALARMS   // 截断尾部旧告警
  }
  if (alarm.level !== 0) {
    this.activeRegions[alarm.region_id] = 'red'
  }
},
```

**后端 `alarms.py:116`**：历史查询加 LIMIT，避免一次性返回全量：

```python
records = (
    query.order_by(AlarmEvent.created_at.desc())
         .limit(200)
         .all()
)
```

> 效果：el-table 渲染行数封顶，push 的响应式 diff 成本恒定，**不再"越直播越卡"**。
> 完整历史仍可在专门的告警历史页分页查询，看板只保留最近若干条。

---

### P1 — drawOverlay 只在必要时重绘

当前 `watch(flashOn)`（`:309`）触发的重绘，在 P0-A 后仅在有红区时发生，已大幅减少。
可进一步：`drawOverlay` 内提前 return 无区域的情况（已有 `:175`），无需额外改。

若区域非常多（>50），再考虑：把 region 静态边框画到一层离屏 canvas 缓存，
只有红闪的填充色每帧重画——非必需，视区域数量再定。

---

### P2 — 视频尺寸抖动防"重复 emit"（次要）

`VideoPlayer.vue:88` 每帧比较尺寸，正常不触发；若后端帧尺寸偶发波动会反复
`emit('dimensions')` → 父组件 `updateOverlaySize` 重绘。可加一次性判断：
尺寸只在真正变化时 emit（当前逻辑已是条件判断，观察 P0 修复后是否仍有抖动再决定）。

## 4. 验证方法

修复前后各录一次 Chrome Performance（5~10s，正常操作 + 制造告警）：

1. **顶部 CPU 条**：紫(Rendering)/黄(Scripting)大片 → 修复后应明显变空。
2. **底部 Summary 饼图**：Rendering + Scripting 占比应下降。
3. **主线程火焰图**：`drawOverlay`、el-table 渲染的宽条应变窄、变稀。
4. **主观**：无告警时界面顺滑；连续产生多条告警后，界面不再逐渐变卡。

DoD：无告警静置时 CPU 条基本空白；持续告警 10 分钟后界面流畅度无明显衰减。

---

## 5. 落地顺序

先做 **P0-A + P0-B**（改动小、直击黄紫两大主因），录一次 Performance 对比确认，
再视情况做 P1/P2。P0 两项互相独立，可分别提交、分别验证。
