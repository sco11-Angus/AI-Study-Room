# 联合调试计划（Joint Debugging Plan）

> 智慧自习室 AI 管家 · 端-流-云-网全链路联调
> 版本 V1.0.0 ｜ 目标：在集成环境中打通视频流 → AI 推理 → 告警闭环 → 前端看板的完整链路

---

## 1. 目标与范围

联合调试的目标是把各成员独立开发、单测通过的模块，在**真实集成环境**中拼接成一条可运行的链路，验证跨模块契约（数据格式、接口地址、WebSocket 消息、告警事件）在联调时不出错。

**范围（四层）**

| 层级 | 组件 | 联调关注点 |
| --- | --- | --- |
| 端 | IP 摄像头 / OBS 推流 | 能否稳定推出 H.264 流 |
| 流 | Nginx-RTMP（`49.233.71.82:9090`） | 推拉流转发、前端可拉流 |
| 云 | Flask + YOLO + Dlib（`:5000`） | 解码、跳帧推理、几何判定、告警状态机、持久化 |
| 网 | Vue3 + Element Plus（Vite dev `:5173`） | 看板、Canvas 画区、参数配置、实时告警 |

**不在本次联调范围**：模型精度调优、生产级性能压测、CI/CD 流水线（见 `Jenkinsfile` 另行处理）。

## 5. 端到端联调场景（核心）

按功能优先级逐场景走通。每个场景记录：**触发方式 → 期望链路 → 实际结果 → 证据**。

### 场景 A：视频流上屏
- 触发：前端选中 camera，播放视频
- 期望：`/ws/video_feed/<id>` 推帧，画面延迟 < 2s
- 证据：截图 + 延迟观测

### 场景 B：危险区域 / 座位防区入侵
- 触发：Canvas 画 ROI，设 `X_distance`/`Y_stay_time`；人员进入防区
- 期望：`IntrusionPlugin` 判定 → `AlarmEvent(type=intrusion/occupy)` → 持久化 → `/ws/alarms` 推送 → 看板对应区域红色闪烁 → 钉钉卡片
- 证据：`GET /api/alarms?status=notified` 有记录 + snapshot 文件 + 钉钉 `errcode=0`

### 场景 C：座位身份核验（占座）
- 触发：`seat_status` 中座位 `studying` 且绑定 `user_id`；非本人进入
- 期望：人脸比对不符 → `AlarmEvent(type=occupy, extra.kind=unauthorized_seat)`
- 前置：`member` 表需有与 `seat_status.user_id` 对应的已录入人脸特征
- 证据：告警记录含 `expected_user_id` / `actual_face_match`

### 场景 D：疲劳检测（学习陪伴）
- 触发：座位切 `studying`，模拟闭眼/打哈欠（EAR/MAR）
- 期望：弱提醒（`level=0` 私有推送，不进群告警）
- 证据：`/ws/face_recognition` 或私有通道消息

### 场景 E：烟火检测
- 触发：向流中注入烟/火画面（30 帧窗口平均置信度 > 45%）
- 期望：`FireSmokePlugin` → `AlarmEvent(type=fire_smoke)` → 告警闭环
- 前置：`backend/model_weights/fire_smoke.pt` 存在，`FIRE_SMOKE_LEGACY_YOLOV5_DIR` 指向 legacy yolov5 目录
- 证据：告警记录 + confidence
- ⚠️ 已知 blocker：缺真实带烟火/反光负样本的 RTMP 素材，需现场或 OBS 补齐

### 场景 F：打架检测
- 触发：模拟肢体冲突（复用入侵检测的全身人体框 shared_ctx）
- 期望：`AlarmEvent(type=fight)`
- 证据：告警记录

### 场景 G：告警闭环 + 钉钉升级
- 触发：任一真实告警产生
- 期望：抓帧裁脸 → 看板闪烁 → 钉钉 ActionCard → 3 分钟未确认自动升级负责人
- 证据：`notification_log` 有 primary + escalated 两条；点击"确认"按钮回写状态；`PUBLIC_BASE_URL` 为 HTTP 而非 RTMP

### 场景 H：违规抓拍回放 + AI 日报（Task G）
- 触发：产生若干告警后查看回放与日报
- 期望：抓拍/片段可回放，日报可生成
- 证据：回放 URL 可访问

## 6. 联调检查清单（跨模块契约）

联调最容易翻车的是接口契约不一致，逐项核对：

- [ ] **AlarmEvent 字段**：`type / region_id / camera_id / ts / level / snapshot_url / face_match / extra` 前后端一致
- [ ] **camera_id 一致性**：scheduler 注册的 id = 数据库 camera.id = 前端选中 id（历史踩过坑）
- [ ] **WebSocket 消息格式**：`/ws/alarms` 推送体字段与前端 AlarmPanel 解析一致
- [ ] **告警去重**：同 region+type 30s 内去重，联调时不要误判"没触发"
- [ ] **level 语义**：`level=0` 私有（疲劳），不进钉钉群
- [ ] **快照路径**：`/api/alarms/{id}/snapshot` 能取到 `backend/snapshots/` 下文件
- [ ] **钉钉 URL**：`PUBLIC_BASE_URL` 必须 HTTP/HTTPS，确认按钮才可点

## 8. 已知风险与 Blocker

- **云端 RTMP 依赖外部状态**：`rtmp://49.233.71.82:9090/live/test` 历史多次 `opened=False`。联调排期需确认云端流在线，或准备本地 OBS/摄像头兜底。
- **烟火真实验收素材缺失**（`task-c3` 仍 blocked）：代码与单测已通过，但缺带烟火 + 反光负样本的真实视频。
- **座位身份核验需 member 数据**：`member` 表须有与 `seat_status.user_id` 对应的人脸特征，否则只能验证"陌生人告警"分支。
- **模型权重体积大且 gitignored**：`fire_smoke.pt`、dlib 权重需在联调机器上预先就位。
- **中文路径构建坑**：直接在中文路径下 `npm.cmd run build` 曾失败，须从仓库根用 `--prefix frontend`。
- **stale 测试**：`test_face.py` 3 例、`test_liveness.py` 1 例断言已过时（非功能 bug），联调判定成败以行为/证据为准，不以这 4 个红灯为准。
