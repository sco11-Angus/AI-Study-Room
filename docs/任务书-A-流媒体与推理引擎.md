# A 任务书 — 流媒体 + 统一推理引擎 + 部署

> 角色定位：项目地基。第一周需最先冻结「帧接口」与「Detector 插件接口」，B/C/D/E/F 都依赖你的产出。
> 关联设计：系统设计说明书 §2、§3、§10、§11。

## 一、职责范围

1. **流媒体（流）**
   - 部署调通 Nginx-RTMP（`streaming/nginx.conf`）。
   - 摄像头 RTMP 推流 → HTTP-FLV，供前端 F 低延迟播放。
2. **拉流与跳帧调度（§3）**
   - 完善 `backend/app/stream/scheduler.py`：拉流解码、有界环形缓冲（丢旧帧）、每 `SKIP_N=5` 帧取 1 帧推理。
   - 断流自动重连。
3. **统一推理引擎（核心，杜绝算力争抢）**
   - 定义 `Detector` 插件接口，维护**唯一**的推理线程池。
   - B/C/D 只提供检测类注册进来，禁止各自 new 线程/加载独立推理循环。
   - 按帧统一派发给已注册检测器，收集其产出的告警事件交给 E。
4. **部署（§11）**
   - `deploy/docker-compose.yml` 调通，流分发与算法后端进程解耦。

## 二、交付物

| 交付 | 说明 | 时间要求 |

| `Detector` 插件接口 | `detect(frame, ts) -> list[AlarmEvent]`，注册/注销 API | **第一天冻结** |
| 帧接口 | 环形缓冲取帧、帧时间戳约定 | **第一天冻结** |

```python
        ...
```

## 四、验收标准

- [ ] Web 端拉流画面整体延迟 ≤ 2s。
- [ ] 2 核 2GB 机器上流分发 CPU < 40%。
- [ ] 三个检测器共用一个线程池，无各自起循环的情况。
- [ ] 断流后可自动恢复。

## 五、协作接口

- **上游**：摄像头 / F（前端播放地址）。
- **下游**：B/C/D（注册检测器）、E（接收告警事件）。
