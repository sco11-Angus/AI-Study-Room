# 协作顺序与联调计划

> 六人分工任务书见同目录 `任务书-A~F`。本文件说明**先做什么、谁等谁、什么时候联调**，避免互相卡壳。
> 关联：系统设计说明书 §13 需求追溯矩阵。

## 一、人员与模块速查

| 人 | 模块 | 分值/权重 |
| --- | --- | --- |
| A | 流媒体 + 统一推理引擎 + 部署 | 地基 |
| B | 防区入侵检测 | 25 分核心 |
| C | 疲劳检测 + 自习伴侣 + 人脸识别 | 模块一 |
| D | 烟火检测 + 数据层 + 接口框架 | 模块三 + 横向 |
| E | 告警中心 + 钉钉闭环 | 加分项 |
| F | 前端全包 | 前端 |

## 二、依赖关系图

```
        A(帧+Detector接口) ──┬──▶ B 入侵检测 ──┐
                            ├──▶ C 疲劳/人脸 ──┼──▶ E 告警中心 ──▶ F 前端大屏
                            └──▶ D 烟火 ──────┘         ▲
                                                        │
   D(数据层/接口) ──────────────────────────────────────┘
   E(告警事件结构) ──────────────────▶ B/C/D 遵循 + F 渲染
```

核心结论：**A、D、E 三个「接口契约」必须最先冻结**，B/C/F 才不会返工。

## 三、三个阶段

### 阶段一：接口冻结（第一天，全员在场）
必须当天敲定并写进代码/文档，之后不轻易改：

1. **A** 交付 `Detector` 插件接口 + 帧接口（取帧、时间戳）。
2. **E** 交付统一告警事件结构（type/region_id/snapshot/face_match/level…）。
3. **D** 交付数据表结构 + API 返回结构 `{code,message,data}`。

> 这三项没定完，其他人先别写业务逻辑，可先搭骨架/环境。

### 阶段二：并行开发（第一周~中期）
接口冻结后各自推进，互不阻塞：

- **A**：Nginx-RTMP 跑通 → 可用帧流（B/C/D 的输入，尽早出）。
- **D**：数据表建好可用（B/C/E 落库依赖，优先做）。
- **B / C / D**：各自实现检测器，包装成 Detector 插件注册进 A 的引擎。
- **E**：告警服务 + 钉钉 + WebSocket，用 mock 告警先自测。
- **F**：用 mock 数据搭三个页面骨架，等接口好了替换。

### 阶段三：联调闭环（中后期）
按告警链路顺序打通：

1. A 帧流 → B/C/D 检测器出结果（先单模块验证）。
2. B/C/D 告警 → E 告警中心（抓拍 + 去重 + 状态机）。
3. E → F 前端（WebSocket 红闪蜂鸣 + 确认）。
4. E → 钉钉（卡片 + 3min 超时升级 + 确认回调）。
5. 全链路压测：延迟 ≤ 2s、CPU < 40%（A 主导）。

## 四、关键协作约束（红线）

1. **无算力争抢**：B/C/D 一律不得自建线程池，只能注册进 A 的唯一推理引擎。
2. **告警结构统一**：所有告警走 E 定义的同一结构，前端只认这一种。
3. **数据层先行**：D 的表结构第一周必须可用，否则 B/C/E 无法落库。
4. **前端后动细节**：F 等 A/D/E 契约冻结后再写联调部分，前期只搭骨架。

## 五、Git 协作规范（§10.3）

- 每人独立 Feature 分支（如 `feat/intrusion`、`feat/frontend`），从 `main` 切出。
- 提交信息清晰（`feat:` / `fix:` / `docs:`），关联模块。
- 合并前自测通过，B 的入侵检测须保证 `tests/test_intrusion.py` 全绿。
- 联调阶段每日合并一次到 `main`，避免大冲突。
