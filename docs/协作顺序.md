# 协作顺序与联调计划

> 六人分工任务书见同目录 `任务书-A~F`。本文件说明**先做什么、谁等谁、什么时候联调**，避免互相卡壳。
> 关联：系统设计说明书 §13 需求追溯矩阵。

## 一、人员与模块速查

| 人 | 模块 | 分值/权重 |
| --- | --- | --- |
| A | 流媒体 + 统一推理引擎 + 部署 | 地基 |
| B | 防区入侵(25分核心) + 疲劳/自习伴侣 + 人脸识别 | 视觉主力（最重） |
| C | 烟火检测 + 数据层 + 接口框架 | 模块三 + 横向支撑 |
| D | 音视频融合语义判断打架 | 技术亮点（新模块） |
| E | 告警中心 + 钉钉闭环 | 加分项 |
| F | 前端全包 | 前端 |

> 任务书文件：`任务书-A-流媒体与推理引擎.md`、`任务书-B-入侵+疲劳+人脸.md`、`任务书-C-烟火+数据层+接口.md`、`任务书-D-音视频打架检测.md`、`任务书-E-告警中心与钉钉闭环.md`、`任务书-F-前端.md`。

## 二、依赖关系图

```
                    ┌──▶ B 入侵/疲劳/人脸 ──┐
 A(帧+音轨          │        │(人员框)        │
   +Detector接口)──┼──▶ C 烟火 ─────────────┼──▶ E 告警中心 ──▶ F 前端大屏
                    │        ▲                │        ▲
                    └──▶ D 打架(音视频融合)──┘        │
                             ▲   ▲                     │
                       复用B人员框 A音轨               │
   C(数据层/接口/返回结构) ───────────────────────────┘
   E(告警事件结构 AlarmEvent) ──────▶ B/C/D 遵循 + F 渲染
```

核心结论：**A、C、E 三个「接口契约」必须最先冻结**（A=帧/音轨/Detector，C=数据层/返回结构，E=告警结构），B/D/F 才不会返工。D 额外依赖 A 的音轨与 B 的人员框。

## 三、三个阶段

### 阶段一：接口冻结（第一天，全员在场）
必须当天敲定并写进代码/文档，之后不轻易改：

1. **A** 交付 `Detector` 插件接口 + 帧接口（取帧、时间戳）+ 音轨抽取边界约定。
2. **E** 交付统一告警事件结构（type/region_id/snapshot/face_match/level/extra…，含 `fight` 类型）。
3. **C** 交付数据表结构（8 张，见 `数据库设计.md`）+ API 返回结构 `{code,message,data}`。

> 这三项没定完，其他人先别写业务逻辑，可先搭骨架/环境。

### 阶段二：并行开发（第一周~中期）
接口冻结后各自推进，互不阻塞：

- **A**：Nginx-RTMP 跑通 → 可用帧流 + 音轨（B/C/D 的输入，尽早出）。
- **C**：数据表建好可用（B/D/E 落库依赖，优先做）+ 接口框架 + 烟火检测。
- **B**：人员检测 + 入侵/疲劳/人脸（**人员框需暴露给 D 复用**）。
- **D**：音轨管线 + 视觉冲突 + 音频打斗 + 融合（依赖 A 音轨、B 人员框）。
- **E**：告警服务 + 钉钉 + WebSocket，用 mock 告警先自测。
- **F**：用 mock 数据搭页面骨架，等接口好了替换。

### 阶段三：联调闭环（中后期）
按告警链路顺序打通：

1. A 帧流/音轨 → B/C/D 检测器出结果（先单模块验证）。
2. B/C/D 告警 → E 告警中心（抓拍 + 去重 + 状态机）。
3. E → F 前端（WebSocket 红闪蜂鸣 + 确认；打架/烟火高优先）。
4. E → 钉钉（卡片 + 3min 超时升级 + 确认回调）。
5. 全链路压测：延迟 ≤ 2s、CPU < 40%（A 主导）。

## 四、关键协作约束（红线）

1. **无算力争抢**：B/C/D 一律不得自建线程池，只能注册进 A 的唯一推理引擎。
2. **人员框只算一次**：D 的打架检测复用 B 的人员检测框，不得重复加载 YOLO；A 需在引擎共享上下文里暴露 B 的检测结果给 D。
3. **音轨只解一次**：D 的音轨来源与 A 定死（A 给字节流 or D 读同一 RTMP 音轨），避免重复解码。
4. **告警结构统一**：所有告警走 E 定义的同一 `AlarmEvent`（含 `fight`/`extra`），前端只认这一种。
5. **数据层先行**：C 的表结构第一周必须可用，否则 B/D/E 无法落库。
6. **前端后动细节**：F 等 A/C/E 契约冻结后再写联调部分，前期只搭骨架。

## 五、Git 协作规范（§10.3）

- 每人独立 Feature 分支（如 `feat/intrusion`、`feat/fight`、`feat/frontend`），从 `main` 切出。
- 提交信息清晰（`feat:` / `fix:` / `docs:`），关联模块。
- 合并前自测通过，B 的入侵检测须保证 `tests/test_intrusion.py` 全绿。
- 联调阶段每日合并一次到 `main`，避免大冲突。
