# D 任务书 — 烟火检测 + 数据层 + 接口框架

> 角色定位：烟火检测本身较轻，配上数据层与接口框架横向支撑，补齐工作量。
> 关联设计：系统设计说明书 §6、§8、§9、§10.3。

## 一、职责范围

1. **烟火检测（模块三，§6）**
   - 加载火灾/烟雾 YOLO 轻量权重（类别 `fire`/`smoke`）。
   - 复用 `detectors/fire_smoke.py` 的 30 帧滑动窗口加权：连续 30 帧平均置信度 > 45% 才判定有效灾情。
   - 包装成 A 的 `Detector` 插件注册，**不自建线程池**。
2. **数据层（§8）**
   - 完善 `models/entities.py` 的 6 张表，编写 DB 初始化脚本。
   - 提供各模块的落库能力（region / seat_status / alarm_event / member / notification_log）。
3. **接口框架（§9、§10.3）**
   - Swagger/OpenAPI 文档完善，统一返回结构 `{code,message,data}`，CORS。
   - 各 API 蓝图接入真实数据库读写。

## 二、交付物

| 交付 | 说明 | 时间要求 |
| --- | --- | --- |
| 数据表 + 初始化 | 6 张表建好可用 | **第一周，B/C/E 依赖** |
| 烟火检测插件 | 30 帧加权，产出灾情告警事件 | 中期 |
| Swagger 文档 | `/apidocs` 可访问，接口齐全 | 联调阶段 |

## 三、验收标准

- [ ] 连续 30 帧均值 > 45% 才告警，光线瞬变不误报。
- [ ] 6 张表结构与 §8 一致，各 API 落库正常。
- [ ] Swagger 文档完整，返回结构统一。

## 四、协作接口

- **上游**：A（帧 + Detector 接口）。
- **下游**：全体（数据层是公共依赖）、E（灾情告警事件）。
