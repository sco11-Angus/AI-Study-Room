## 1. 概述

### 1.1 文档目的

本文档面向研发、测试与运维人员，将 PRD 中的产品需求转化为可落地的技术方案，明确系统的整体架构、模块划分、核心算法、数据模型、接口契约与部署拓扑，作为编码实现、联调测试与验收的技术依据。

### 1.2 设计目标

- **实时性**：拉流端到端画面延迟 ≤ 2s；AI 推理通过跳帧机制跟上流媒体帧率。
- **准确性**：通过时空防抖与连续帧置信度加权，抑制光线瞬变、遮挡等引起的误报。
- **可配置**：区域防区、安全距离、停留时间等业务参数支持前端可视化配置并持久化。
- **闭环性**：异常事件从检测 → 抓拍 → 看板告警 → 钉钉逐级上报形成完整闭环。
- **低成本**：可在 2 核 2GB、4Mbps 带宽的低配 Linux 云服务器上稳定运行。

### 1.3 术语表

| 术语 | 说明 |
| --- | --- |
| ROI | Region of Interest，前端勾勒的多边形防区 |
| EAR | Eye Aspect Ratio，眼睛长宽比，用于闭眼判定 |
| MAR | Mouth Aspect Ratio，嘴巴长宽比，用于打哈欠判定 |
| 基准点 | 人员检测框底边中心 `(cx, cy)`，近似双脚着地坐标 |
| 时空防抖 | 同时满足空间条件（距离）与时间条件（停留时长）才触发告警 |
| 跳帧 | 每 N 帧仅取 1 帧送入 AI 推理，其余帧仅转发显示 |

## 3. 流处理与调度设计（性能基石）

### 3.1 拉流与解码

- 后端通过 OpenCV `VideoCapture` 或 FFmpeg 从 Nginx-RTMP 拉取 RTMP/FLV 流。
- 每路摄像头对应一个独立的解码线程，将解码帧写入固定长度的**环形帧缓冲队列**（丢弃最旧帧，保证低延迟不堆积）。

### 3.2 跳帧调度机制

为使 AI 推理速度匹配流媒体帧率，采用“显示全帧、推理跳帧”策略：

- **显示链路**：所有帧原样交由 Nginx-RTMP → 前端播放，保证画面流畅、延迟 ≤ 2s。
- **推理链路**：调度器每 `SKIP_N = 5` 帧取 1 帧送入算法线程池执行几何/目标检测。
- 推理结果（检测框、告警状态）以叠加层形式回写，前端按最近一次结果渲染。

```python
frame_idx = 0
while running:
    frame = capture.read()            # 解码线程
    ring_buffer.push(frame)           # 供推流/显示
    if frame_idx % SKIP_N == 0:
        infer_pool.submit(analyze, frame)   # 仅每5帧推理一次
    frame_idx += 1
```

### 3.3 性能约束落实

| 指标 | 目标 | 设计手段 |
| --- | --- | --- |
| 画面延迟 | ≤ 2s | 环形缓冲丢旧帧 + HTTP-FLV/低延迟 HLS |
| 流分发 CPU | < 40%（2 核） | 转发与推理进程分离；推理跳帧；YOLOv8n 轻量模型 |
| 推理不堆积 | — | 有界队列 + 丢帧策略，禁止无限积压 |

## 4. 模块一：用户自习状态宣告（自习伴侣）

### 4.1 设计要点

用户状态直接决定疲劳检测算法对该座位区域的**激活 / 挂起**，实现“不打扰”与算力释放。

### 4.2 状态机

```
        切换"自习中"(studying)                切换"休息中"(resting)
 ┌──────┐ ───────────────────▶ ┌──────────┐ ───────────────────▶ ┌──────────┐
 │ 空闲  │                      │ 自习中     │                      │ 休息中     │
 │ idle │ ◀─────────────────── │ studying │ ◀─────────────────── │ resting  │
 └──────┘   离座/预约结束        └──────────┘   切回自习中           └──────────┘
                                 疲劳算法：激活                       疲劳算法：挂起(释放CPU)
```

### 4.3 疲劳检测算法（迁移自疲劳驾驶）

- 使用 Dlib 68 点人脸模型定位眼、嘴关键点。
- **EAR（眼睛长宽比）**：
  `EAR = (||p2-p6|| + ||p3-p5||) / (2 * ||p1-p4||)`
  当 `EAR < EAR_THRESH` 且**持续 ≥ 2s**，判定为闭眼睡觉。
- **MAR（嘴巴长宽比）**：超过 `MAR_THRESH` 判定为打哈欠。
- **提醒方式**：仅向该用户专属屏幕/手机推送**弱提醒**，不触发公用警报。

### 4.4 行为规则

| GIVEN | WHEN | THEN |
| --- | --- | --- |
| 用户已预约入座 | 状态切为 `studying` | 激活该座位疲劳检测算法 |
| 疲劳算法运行中 | EAR 低于阈值持续 2s / MAR 超标 | 推送弱提醒（私有，不拉公用警报） |
| 用户处于自习 | 状态切为 `resting` | 立即挂起该区域疲劳算法，释放 CPU，趴桌/闭眼均不干扰 |

## 5. 模块二：前端交互画区 + 危险区域/座位防区检测（核心）

### 5.1 前端 Canvas 交互设计

- 在播放的视频流上叠加透明 `<canvas>` 图层（与视频同尺寸、坐标对齐）。
- 交互流程：
  1. 鼠标单击逐个添加多边形控制点，实时连线预览。
  2. 双击闭合，生成一个多边形 ROI。
  3. 支持在同一画面创建多个防区/独立座位格子，可命名、删除、重绘。
- **坐标归一化**：Canvas 顶点坐标按显示尺寸归一化为 `[0,1]`，提交后端时映射回原始分辨率，避免不同分辨率错位。

### 5.2 可配置参数

| 参数 | 含义 | 单位 |
| --- | --- | --- |
| `X_distance` | 安全距离阈值 | 像素 |
| `Y_stay_time` | 允许危险停留时间 | 秒 |
| `polygon` | 多边形顶点序列 | 像素坐标 |

### 5.3 后端几何判定算法

- YOLOv8n 输出人员检测框，取**底边中心点** `(cx, cy)` 作为判定基准点（近似双脚坐标）。
- 调用 `cv2.pointPolygonTest(polygon, (cx, cy), True)` 得到带符号最短距离 `D`：

```
D = cv2.pointPolygonTest(polygon, (cx, cy), measureDist=True)

判定优先级：
  ① D >= 0                      → 已闯入区域    → 危险计时器持续累计
  ② D < 0 且 |D| <= X_distance   → 低于安全距离   → 危险计时器持续累计
  ③ 其余                         → 安全          → 危险计时器清零
```

### 5.4 时空防抖状态机

```
        进入危险(①或②)               持续危险且累计 >= Y_stay_time
 ┌──────┐ ──────────────▶ ┌──────────┐ ──────────────────────────▶ ┌──────────┐
 │ 安全  │                 │ 计时中     │                             │ 触发告警   │
 │ SAFE │ ◀────────────── │ COUNTING │                             │ ALARM    │
 └──────┘   离开危险区,清零  └──────────┘                             └──────────┘
```

- 计时器要求**无间断累计**：任意一帧回到安全状态即清零，防止抖动误报。
- 计时以“时间”为单位而非帧数，兼容跳帧场景（用两次有效推理的时间戳差累加）。
- 触发后进入 `ALARM`，向告警中心投递事件（详见模块四）。

## 6. 模块三：环境异常识别（烟雾与明火检测）

### 6.1 模型

- 集成火灾/烟雾专用轻量化目标检测模型（基于 YOLO 权重优化，类别：`fire`、`smoke`）。
- 与人员检测复用同一跳帧推理链路，降低整体负载。

### 6.2 连续帧置信度加权防误报

只有同时满足以下条件才判定为**有效灾情**：

- 在**连续 30 帧**（指进入推理的有效帧）中，`fire`/`smoke` 类别的**平均置信度 > 45%**。

```
滑动窗口 W = 最近30次有效推理的置信度
若 mean(W) > 0.45 且 窗口已满  → 判定有效灾情 → 触发告警
任一帧未检出/低置信         → 计入窗口(以0或实际值)，均值自然下降 → 抑制瞬变误报
```

- 该机制过滤反光、瞬时光线变化等短时干扰，避免单帧高置信误触发。

## 7. 模块四：告警中心与钉钉逐级闭环通知（加分项）

### 7.1 闭环动作总览

```
真实告警产生
   │
   ├─▶ ① 现场抓拍：抓取当前帧 → 裁剪面部 → 人脸识别匹配(是否注册会员)
   │
   ├─▶ ② 看板异动：对应格子 绿→红闪烁 + 局部蜂鸣 (WebSocket 推送前端)
   │
   └─▶ ③ 钉钉逐级上报：Webhook 卡片消息 → 3分钟未确认 → 升级推送科长/负责人
```

### 7.2 现场抓拍与人脸识别

- 告警瞬间从帧缓冲抓取当前帧并落盘（作为证据），裁剪入侵者面部区域。
- 送 Dlib 人脸特征提取，与注册会员特征库做欧氏距离匹配，输出“会员/陌生人”标签附于告警。

### 7.3 看板异动

- 后端经 WebSocket 向前端推送告警事件，前端将对应防区/座位格子由绿变为**红色闪烁**并触发局部蜂鸣。

### 7.4 钉钉超时升级机制（工作流）

```
        发送卡片给主责安全员
 ┌──────────┐ ───────────▶ ┌──────────────┐  点击"确认处理"  ┌──────────┐
 │  待通知    │              │ 已通知(计时3min)│ ─────────────▶ │ 已处理     │
 └──────────┘              └───────┬───────┘                └──────────┘
                                   │ 3分钟超时未确认
                                   ▼
                          ┌──────────────────┐
                          │ 升级：推送科长/负责人 │
                          └──────────────────┘
```

- 调用钉钉群机器人 Webhook 发送 ActionCard 格式化卡片，含抓拍图、类型、位置、时间、“确认处理”按钮。
- 首位主责安全员须在 **3 分钟**内点击确认；超时由后台工作流自动**升级优先级**并推送至保卫科长/直属负责人。
- 后端提供确认回调接口，收到点击后停止升级计时。

## 10. 非功能性设计

### 10.1 性能与延迟

| 需求 | 设计落实 |
| --- | --- |
| 画面延迟 ≤ 2s | Nginx-RTMP + HTTP-FLV 分发；环形缓冲丢旧帧 |
| 推理跟得上帧率 | 每 5 帧推理一次；YOLOv8n 轻量模型；线程池并发 |
| 流分发 CPU < 40%（2核2GB/4Mbps） | 转发与推理进程解耦；限制并发路数；参数可调 SKIP_N |

### 10.2 可靠性

- 帧队列有界，异常帧跳过不阻塞；拉流断开自动重连。
- 告警去重：同防区同类型在冷却窗口内合并，避免刷屏。

### 10.3 安全与规范

- **接口规范**：全部核心 API 引入 Swagger (OpenAPI)，自动化文档输出，拒绝盲目联调。
- **数据安全**：人脸特征等敏感数据加密存储；抓拍图访问受限；配置文件不入库明文密钥。
- **代码合规**：遵循 OpenSpec 规范驱动，先由 `spec.md` 明确边界再编码；Git 采用 Feature/Bugfix 分支，提交信息清晰可追溯。
