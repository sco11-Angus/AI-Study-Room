## 1\. 引言与背景

### 1\.1 产品背景

传统的共享自习室采用“无人值守”模式，虽然节省了人工成本，但带来了严重的管理痛点：恶意占座、非缴费人员尾随进入（白嫖）、违规喧哗或抽烟影响他人，以及深夜突发人身安全隐患。本系统旨在利用现代“端\-流\-云\-网”全栈架构，引入计算机视觉（CV）算法，将被动的视频监控升级为积极主动的“智能AI管家”。

### 1\.2 产品核心价值

- **空间精细管理**：通过数字化区域划分，解决违规占用、占座等顽疾。

- **秩序智能维护**：全天候感知抽烟、喧哗及疲劳状态，提供人文关怀式的主动提醒。

- **安全防范闭环**：结合物联网与即时通讯（钉钉群机器人），打通异常事件通知的“最后一公里”。

## 2\. 系统架构设计

系统采用“端\-流\-云\-网”一体化架构：

1. **数据采集端（端）**：自习室网络摄像头采集 H\.264/AAC 音视频流。

2. **流媒体服务器（流）**：基于云端自建的 **Nginx\-RTMP** 服务器进行高并发推拉流转发。

3. **算法与业务后端（云）**：Python 后端模块，集成 YOLO 目标检测、Dlib 人脸识别算法、空间距离判定逻辑及持久化数据库。

4. **前端展示端（网）**：Vue 响应式管理看板，支持交互式画区与实时告警中心。

## 3\. 功能需求详述

### 模块一：用户自习状态宣告（自习伴侣）

#### 【功能描述】

允许自习学生自主上报/切换当前的座位使用状态，系统根据状态动态激活或挂起 AI 行为分析算法，实现“不打扰”的人文关怀。

#### 【业务规则与场景 \(Scenario\)】

- **GIVEN（前提条件）**：用户已成功预约并入座。

- **WHEN（触发事件）**：用户在小程序/Web端将状态切换为 `"自习中" (studying)`。

- **THEN（系统行为）**：后端激活针对该座椅区域的**疲劳驾驶检测算法（迁移至自习场景）**。

- **AND**：若通过 Dlib（68点人脸模型）计算其眼睛长宽比（EAR）持续低于阈值 2 秒（闭眼睡觉），或嘴巴长宽比（MAR）超标（打哈欠），系统将在其专属屏幕或手机上推送弱提醒，不拉响公用警报。

- **WHEN（触发事件）**：用户将状态切换为 `"休息中" (resting)`。

- **THEN（系统行为）**：后端**立即挂起**该区域的疲劳识别算法（释放系统 CPU 算力），不论其闭眼还是趴桌，系统均不做干扰。

### 模块二：基于前端交互的危险区域/座位防区检测

#### 【功能描述】

管理员可在 Web 端监控视频上方，利用 Canvas 鼠标交互自由勾勒出多边形安全防区（或独立座位格子）。系统实时计算人员与边缘的相对空间距离，并实现可配置的时空告警防抖。

#### 【页面交互与参数配置】

1. **防区配置页面**：用户在播放的视频流图层上，通过鼠标点击建立多个控制点，双击闭合生成一个多边形 ROI 区域。

2. **配置项持久化**：用户可提交并配置参数：

- `X_distance`（安全距离阈值，单位：像素）

- `Y_stay_time`（允许危险停留时间，单位：秒）

#### 【后端几何算法与判定逻辑】

- **时空碰撞检测**：后端 YOLOv8n 提取人员检测框，以其底边中心点（双脚坐标）作为判定基准点 $(cx, cy)$。

- **公式判定**：调用 `cv2.pointPolygonTest(polygon, (cx, cy), True)` 计算最短距离 $D$：

1. 若 $D \ge 0$，判定为“已闯入区域” $\rightarrow$ 激活危险计时器。

2. 若 $D < 0$ 且 $|D| \le X\_distance$，判定为“低于安全距离” $\rightarrow$ 激活危险计时器。

3. 若危险计时器无间断累计时间 $\ge Y\_stay\_time$，系统锁定并**触发真实告警事件**。

### 模块三：环境异常识别（烟雾与明火检测）

#### 【功能描述】

自习室内部全封闭场景下，实时监测明火、抽烟产生的烟雾等紧急消防事件。

#### 【技术要求】

- 后端集成火灾烟雾专用轻量化目标检测模型（基于 YOLO 权重优化）。

- **防误报规则**：系统采取连续帧置信度加权算法，只有在**连续 30 帧**视频流中，烟雾/明火类别的平均置信度超过 45% 时，才判定为有效灾情，防止光线瞬变引起误报。

### 模块四：告警中心与第三方闭环通知

#### 【功能描述】

当上述模块（非法闯入、烟雾、恶意占座）产生真实告警时，系统需立刻实现“云\-网”闭环联动。

#### 【系统闭环动作】

1. **现场抓拍**：后端立刻抓取当前帧视频图像，截取入侵者面部发送至人脸识别模块进行特征匹配（识别是否为注册会员）。

2. **看板异动**：Web端监测大屏对应的座位/防区格子由绿色瞬间变为红色闪烁，并伴随局部蜂鸣。

3. **钉钉逐级上报 \(CI/CD 规范应用\)**：

- 调用**钉钉群机器人 Webhook API** 发送格式化卡片消息。

- **超时升级机制**：首位安全员（主责）接收到通知后，必须在 3 分钟内点击卡片进行“确认处理”。若超时未响应，系统基于工作流自动将该告警提升优先级，推送到保卫科长或直属负责人的钉钉上。

## 4\. 非功能性需求



### 4\.1 性能与延迟需求



- **算法跳帧优化**：为防止 AI 推理速度跟不上流媒体帧率，后端必须采用跳帧处理机制（每 5 帧丢给 YOLO 算一次空间几何），确保 Web 端拉流**画面整体延迟控制在 2 秒以内**。

- **服务器负载**：在 2核\-2GB 内存、4Mbps 带宽的低配 Linux 云服务器上，系统流分发占用的 CPU 需低于 40%。

    

### 4\.2 安全与规范性



- **接口规范**：系统所有后端核心 API 必须引入 **Swagger \(OpenAPI\)** 规范进行自动化文档输出，拒绝盲目联调。

- **代码合规**：开发过程必须严格遵循 **OpenSpec** 规范驱动（由 `spec.md` 明确边界后编写代码），Git 提交必须有清晰的 Feature/Bugfix 分支线。
