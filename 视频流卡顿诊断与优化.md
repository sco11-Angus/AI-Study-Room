# 视频流卡顿诊断与优化

> 针对前端监测大屏视频流卡顿问题的链路排查与修复方案
> 分析范围：解码线程 → ring_buffer → `/ws/video_feed` 推送 → 前端 Canvas 渲染
> 说明：本文基于**静态代码分析**，主因判断为推断，需按 §5 实测信号确认。

---

## 1. 链路概览

```
RTMP(49.233.71.82:9090)
   └─ 解码线程 _decode_loop  (backend/app/stream/scheduler.py)
        ├─ cap.read() 拉帧 → 等比缩放(TARGET_W=640) → cv2.imencode JPEG(q=65)
        ├─ ring_buffer(maxlen=5) 追加 → notify_all()   ← 显示链路
        └─ 每 SKIP_N=5 帧 → engine.dispatch_async()     ← 推理链路
   └─ /ws/video_feed/<id>  (backend/app/api/video_feed.py)
        └─ wait_frame() → latest_frame() → ws.send(jpg)  限 _MAX_FPS=12
   └─ 前端 VideoPlayer.vue
        └─ onmessage(Blob) → new Image() → drawImage(canvas) 限 15fps
```

---

## 2. 根因分析（按影响从大到小）

### 2.1 单线程推理 + 开发服务器阻塞（最可能主因）

- `backend/run.py:77` 使用 `app.run()`（Werkzeug 开发服务器），**未开 `threaded=True`**。
  默认情况下并发请求会互相阻塞，而看板同时开了 3 条长连接 WebSocket
  （`/ws/video_feed`、`/ws/alarms`、`/ws/face_recognition`），彼此争用会造成卡顿。
- `backend/app/stream/engine.py:48` 推理线程池 **`max_workers=1`**（注释写 2，实际默认 1）。
  5 个检测器（intrusion / face / fatigue / fire_smoke / fight，含 YOLO + dlib）
  **串行挤在一个线程**执行。
- `scheduler.py:349` 每 5 帧 `dispatch_async` 提交一帧到线程池。当推理耗时超过帧间隔，
  `ThreadPoolExecutor` 任务队列**无限堆积**，延迟持续累积，画面表现为一顿一顿。

### 2.2 前后端帧率不一致

| 位置 | 限制 | 值 |
| --- | --- | --- |
| 后端推送 | `video_feed.py:26` `_MAX_FPS` | 12 fps |
| 前端渲染 | `VideoPlayer.vue:23` `FRAME_INTERVAL` | 15 fps |

前端想要 15fps、后端只给 12fps，节奏错配。叠加 `ring_buffer(maxlen=5)` 的
「丢旧帧保低延迟」策略，一旦解码变慢，前端收到的就是跳跃的非连续帧。

### 2.3 前端每帧解码开销（次要）

`VideoPlayer.vue:75-90` 每帧 `new Image()` + `createObjectURL` + 异步 `onload` 再 `drawImage`。
高分辨率下异步解码链有开销，帧到得比解码快时会堆积、产生乱序感。

### 2.4 RTMP 抖动被重试逻辑放大（次要）

`video_feed.py:61` 超时后 `_MAX_RETRIES=2` × `_FRAME_TIMEOUT=1.0s`，
单次最多阻塞 2 秒才回一个 `waiting`。上游 RTMP 抖动时停顿会被放大。

---

## 3. 修复方案（按性价比排序）

### P0 — 让后端能并发（改动小、风险低、最可能见效）

1. **开启多线程 Web 服务**（`backend/run.py:77`）
   ```python
   app.run(host="0.0.0.0", port=port, debug=False, threaded=True)
   ```
   避免 3 条 WebSocket 长连接互相阻塞。

2. **扩大推理并发**（`backend/run.py:28`）
   ```python
   engine = InferenceEngine(max_workers=2)  # 或按 CPU 核数调
   ```

3. **给 `dispatch_async` 加背压**（`backend/app/stream/engine.py:109`）
   队列满就丢当前帧的推理（显示链路不受影响），避免无限堆积：
   ```python
   def dispatch_async(self, frame: Frame) -> None:
       if not self._detectors:
           return
       # 池繁忙时丢帧，防止延迟累积
       if self._pool._work_queue.qsize() > 2:
           return
       self._pool.submit(self._dispatch_and_raise, frame)
   ```

### P1 — 对齐帧率

后端 `_MAX_FPS`（`video_feed.py:26`）与前端 `FRAME_INTERVAL`（`VideoPlayer.vue:23`）
统一到同一值（建议都 15 或都 12），消除错配。

### P2 — 前端解码优化

`VideoPlayer.vue:75-90` 用 `createImageBitmap(blob)` 替代 `new Image() + createObjectURL`，
减少每帧解码开销、避免乱序：
```javascript
ws.onmessage = async (event) => {
  if (!(event.data instanceof Blob)) { /* ...JSON 状态处理... */ return }
  const now = Date.now()
  if (now - lastFrameTime < FRAME_INTERVAL) return
  lastFrameTime = now
  const bitmap = await createImageBitmap(event.data)
  // canvas 尺寸自适应 + drawImage(bitmap) ...
  bitmap.close()
}
```

### P3 — RTMP 抖动容忍

视情况下调 `_FRAME_TIMEOUT` 或 `_MAX_RETRIES`（`video_feed.py:22-24`），
在「卡顿」与「误报缓冲中」之间取平衡。

## 5. 验证方法（确认主因）

起后端后观察两个信号：

1. **解码掉帧率**：日志 `[scheduler] cam-x 解码统计: ok=.. dropped=.. 掉帧率=..%`
   （`scheduler.py:358`）。掉帧率高 → 解码/推理端瓶颈。
2. **CPU 占用**：卡顿时若单核打满 → 坐实单 worker 串行推理是瓶颈。

修复后应看到：掉帧率下降、画面延迟稳定 < 2s（对齐《联合调试计划》场景 A 的 DoD）。

---

## 6. 实测结果（2026-07-11）

已落地 P0（后端并发）+ P1（帧率对齐）+ P2（前端 createImageBitmap），并实测验证。

**测试方法**：起真实后端（5 个检测器全注册：intrusion/face/fatigue/fire_smoke/fight）
+ WebSocket 客户端端到端测量。源用本地生成的 720p@30fps 测试视频
（`ffmpeg testsrc`，不触碰共享 RTMP 服务器）。优化前/后各跑 2 次取稳定值。

| 配置 | 实测 FPS | 帧间隔 avg | 帧间隔 max |
| --- | --- | --- | --- |
| 优化前（max_workers=1、无背压、_MAX_FPS=12） | 11.0 | 92ms | 532ms |
| **优化后**（max_workers=2、背压、_MAX_FPS=15） | **13.6** | **74ms** | 511ms |

吞吐 +24%，帧间隔更平滑，稳定复现。

### ⚠️ 实测暴露的关键点：源端帧率才是「1fps 级卡顿」的元凶

最初用**本地摄像头**测，优化前后都只有 **1.0 FPS**。排查确认**不是代码问题**——
这台机器的摄像头硬件裸 `cv2.VideoCapture.read()` 就只出 1fps
（DirectShow 低光自动曝光降帧）。换成 30fps 视频源后管线才跑到 13.6fps。

**联调启示**：现场若出现「视频巨卡（~1fps）」，先确认**上游源
（摄像头 / RTMP 推流）本身的帧率**，不要先怀疑管线代码。管线在 30fps
源下能稳定跑到 13.6fps，本身是健康的。

---

## 7. 结论

卡顿是**多因叠加**，而非单一 bug。

- **管线侧**：P0（多线程 + max_workers 提升 + 背压）+ P1（帧率对齐）实测有效，
  吞吐 11.0→13.6 FPS。但在 CPU 推理这台机器上，单 worker 的瓶颈没有预想的极端，
  背压的价值在于防止延迟无限累积（max 帧间隔 532→511ms）。
- **源端侧**：真正会造成「1fps 级卡顿」的是**上游源帧率**，这是 §2 静态分析未充分
  强调、实测才暴露的点，也是联调排障应优先确认的方向。

P0/P1/P2 已落地并验证。P3（RTMP 抖动容忍）为调参项，视现场情况再定。
